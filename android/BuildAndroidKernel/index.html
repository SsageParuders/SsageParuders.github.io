# 如何刷入内核模块

刷入内核模块的思路大概分为两种

  • 刷入自编译的内核与模块
    自己编译的内核
    甚至你还可以在编译时 选择对应参数
    不开启内核校验 这样任何模块均可刷入
  • 绕过内核签名校验和 crc 校验
    利用 Kernel 的漏洞 绕过内核的校验
    实现该方案的途径并不唯一 有很多漏洞 / 骚操作可以利用
    但是无论如何 使用该方案需要更高的技术要求

# 替换自编译的安卓内核

如果你选择了替换内核的方式刷入内核模块
那么 你有两条方式可以选择

  1. 对模块签名
    即使是各项环境相同 每次编译内核产生的 key 都是不一样的
    使用 key 对模块进行签名 如此也保证了内核安全
    使用自己编译的内核产出的 key 对后续编译的动态模块进行签名
    这样 签名后的模块自然可以刷入到自己编译的对应内核中去
    由于博主没有在小米内核中找到对应的签名 key
    ~~ 可能是我眼浊 没看到 但是我严重怀疑小米的 key 重命名了～～{.danger}
    因此 这里不对这种方式多做说明 (未来有时间可能补上)
  2. 关闭校验
    将自己编译的内核关闭校验
    在对内核源码进行 make 时
    一般都会指定配置文件
    比如小米 6 在 make 的时候 配置文件就是 sagit_user_defconfig
    配置文件中有各项编译参数
    其中有如下三个参数是有关于内核签名的
    • CONFIG_MODULE_SIG:
      模块签名验证功能 配置为 "y" 后 才有如下的子选项可以配置
    • CONFIG_MODULE_SIG_FORCE:
      是否验证模块的签名
      配置为 "n" 时 对于未签名或者使用无效的 key 签名的模块 内核都允许加载
      但是 kernel 会被标记为被感染 (tainted) 并且使用商业许可的模块也会被标记为被感染
      若配置为‘y’ 则只有正确签名的模块才能被允许加载
      其他的模块将被拒绝加载 并且 kernel 会打出错误信息
    • CONFIG_MODULE_SIG_ALL:
      若配置为 "n" 则在 build kernel 的过程中
      所有的模块都不会被签名 需要自己手动签名
      若被配置为 "y" 则在 build kernel 的 "modules_install" 阶段
      所有的模块都会被自动签名
      如上所述 只需要在 make 前 将配置文件中的
      CONFIG_MODULE_SIG_FORCE 参数改为 “n”
      制作出来的内核即可刷入任意模块 不过会被标记
      因此 较为适合开发环境 不适合发行环境
      对我们开发者来说 这项参数为 "n" 即可
